name: multi-role-code-review
description: 写完代码后，以不同角色视角 review 当前代码变更，确保质量。
---

### Basic Rules

- 每次完成代码编写（新功能、重构、修复）后，**必须执行多角色 Review**
- 先获取当前变更：`git diff` 或工作区变更
- 按角色逐一 Review，每个角色视角不同

---

### Execution Steps

**Step 1: 架构师 Review**

视角：系统整体设计、可扩展性、技术债

关注点：
- 变更是否符合现有架构分层（backend/app/ 的模块边界）
- 是否引入了不必要的依赖或重复代码
- 数据库 schema 变更是否有 migration
- 接口设计是否一致（RESTful、请求/响应格式）
- 是否有安全漏洞（SQL 注入、XSS、权限绕过）

**Step 2: 资深工程师 Review**

视角：代码质量、性能、健壮性

关注点：
- 错误处理是否完整（try/catch、异常传播、用户反馈）
- 边界条件（空值、超长输入、并发）
- 性能问题（N+1 查询、不必要的循环、大对象传输）
- 日志是否适当（不要过多也不要过少）
- 测试覆盖（新代码必须有对应测试）
- TypeScript 类型是否正确，避免 `any`

**Step 3: 产品经理 Review**

视角：用户体验、功能完整性、交互一致性

关注点：
- 用户看到什么？（加载态、空态、错误态、成功态）
- 操作流程是否顺畅（最少点击次数）
- 文案是否清晰（中英文都要检查）
- 与现有 UI 风格一致（宋式美学、配色、字体）
- 是否需要文档更新（README、FAQ）

**Step 4: 汇总输出**

格式：
```
## 多角色 Review 报告

### 🏗️ 架构师
- [问题/建议] ...（严重程度: P0/P1/P2）

### 🔧 资深工程师
- [问题/建议] ...

### 🎨 产品经理
- [问题/建议] ...

### ✅ 总结
- 可通过: 无 P0 问题
- 需修复后通过: 有 P0/P1 问题
```
