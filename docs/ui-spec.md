# Agentic Data Analyst — UI 设计规格

## 设计风格：中式宋氏美学

整体视觉参考宋代美学——**极简、雅致、留白、含蓄**，不是现代极简（北欧/日本），而是有深厚文化底蕴的中式审美。

### 视觉元素

| 元素 | 规范 |
|------|------|
| 主色调 | 墨色 (#2c2c2c)、宣纸白 (#f5f0e8)、青瓷绿 (#7ba893)、朱砂红 (#c23a2b 点缀) |
| 背景 | 米白/宣纸色，有细微纸纹质感 |
| 字体 | 标题用宋体/思源宋体，正文用楷体或系统默认 |
| 边框 | 极细墨线（1px #d4c9b8），似毛笔勾勒 |
| 留白 | 大量留白，元素间距宽松 |
| 图标 | 水墨风格或简洁线条图标 |
| 分隔线 | 细长线，类似毛笔扫过效果 |
| 阴影 | 极淡，模仿宣纸叠加的层次感 |
| 圆角 | 极小（2-4px）或直角，不突兀 |

### 色彩具体

```
宣纸白   #f5f0e8 — 主背景
墨色     #2c2c2c — 主要文字
淡墨     #666660 — 次要文字
青瓷     #7ba893 — 强调色（按钮、激活状态）
朱砂     #c23a2b — 仅用于警示/删除
茶色     #d4c9b8 — 边框、分隔线
烟灰     #e8e3d8 — 悬停背景
```

### 设计原则

- **留白是设计的一部分**，不是没做完
- 每个元素都有存在的理由
- 动画克制、自然、不做作
- 信息层次通过大小、粗细、颜色区分，不依赖框和背景色
- 整体感觉像打开一卷古籍，但操作是现代的

## 三栏详细设计

### 左栏：历史会话 + 数据库管理

**分上下两部分：**

**上半部分：历史会话列表**

- 按日期分组（今天、本周、更早）
- 每条会话显示自动生成的标题（从第一条用户问题提取）
- 点击切换会话，聊天窗口加载对应的历史消息
- 当前会话高亮
- 右键/长按可重命名或删除会话
- 会话数据持久化存储（服务端 API）
- 底部显示当前会话信息

**下半部分：数据库管理**

**目标用户：懂数据库的人（老板或他的技术员）**

功能：
- 显示已添加的数据库列表（名称 + 图标）
- 点击数据库切换当前查询目标
- 当前选中的数据库高亮

**"添加数据库"流（点按钮弹窗）：**

```
┌─────────────────────────┐
│  添加数据库              │
│                         │
│  数据库类型:             │
│  [DuckDB  ▼]            │
│                         │
│  名称: [零售数据库]      │
│  文件路径: [./data/shop] │
│  或连接字符串:           │
│  [                      │
│  ]                      │
│                         │
│  表过滤（可选）:          │
│  包含: [fct_*, dim_*]   │
│  排除: [temp_*]         │
│                         │
│  [测试连接] [保存并连接]  │
└─────────────────────────┘
```

数据库类型支持（初期）：
- DuckDB（本地文件）
- SQLite（本地文件）
- PostgreSQL
- MySQL
- ClickHouse
- Snowflake

### 中栏：聊天窗口

**目标用户：老板、业务人员（完全不懂技术）**

- 消息气泡样式（类似微信/WhatsApp）
- 用户消息：深蓝色右对齐
- Agent 回复：白色左对齐
- 查询结果表格自动渲染
- 支持 Markdown 表格、图表（后续）

消息类型：

```
用户: "上月销量最高的产品有哪些？"
Agent: 查到了！上月销量前 3 的产品是：
┌──────────┬──────────┬──────────┐
│ 产品名   │ 销量     │ 销售额   │
├──────────┼──────────┼──────────┤
│ 产品A    │ 1,200    │ 360,000  │
│ 产品B    │ 980      │ 245,000  │
│ 产品C    │ 750      │ 187,500  │
└──────────┴──────────┴──────────┘
要我再看看别的维度吗？比如按区域或按月份对比？
```

输入框下方有快捷示例问题（根据当前数据库自动生成）。

### 右栏：系统配置

**目标用户：老板本人或管理员**

分三个区域：

**1. LLM 配置**
```
LLM 提供商: [OpenAI            ▼]
           OpenAI / Azure OpenAI / Anthropic / Ollama / 自定义
API Key:    [••••••••••••••••••••] (密码输入)
API Base:   [https://api.openai.com/v1] (可自定义)
模型:       [gpt-4o                ▼]
           下拉列表根据提供商动态加载
测试连接:   [🔌 测试] ✅ 已连接
```

**2. 数据安全设置**
```
☑ 只读模式（禁止修改数据的 SQL）
☑ 发送数据到 LLM 前需要我确认
☐ 记录所有查询日志
屏蔽的表: [employee_salary, pii_data]
每页最多: [1000] 行
```

**3. 高级设置**
```
对话语言: [中文 ▼]
温度: [━━━━●━━━━] 0.0 ← → 1.0
会话过期: [60] 分钟
```

## 交互细节

| 操作 | 行为 |
|------|------|
| 拖拽分栏 | 左栏/右栏与中栏之间的分隔线可左右拖拽调整宽度 |
| 切换会话 | 左栏点击历史会话，聊天窗口加载对应消息，上下文恢复 |
| 新建会话 | 点击左栏「＋」或直接在中栏发新消息自动建新会话 |
| 重命名会话 | 右键/长按会话名称可编辑 |
| 删除会话 | 右键/长按可删除，确认后不可恢复 |
| 切换数据库 | 右栏选择数据库，当前会话继续使用新数据源 |
| 修改 LLM 配置 | 实时生效，下次 Agent 调用用新配置 |
| 测试连接（数据库） | 弹窗显示 "✅ 连接成功，发现 15 张表" 或错误信息 |
| 测试连接（LLM） | 发一条简单请求确认 API Key 有效 |
| 右侧收起 | 小屏幕时右侧自动隐藏，加一个展开按钮 |

## 技术偏好

- React + Tailwind CSS（前端主力）
- 所有配置通过 `/api/config` 接口读写
- 所有数据库操作通过 `/api/database/*` 接口
- 所有会话操作通过 `/api/sessions/*` 接口
- 内嵌在 FastAPI 的静态文件目录 `ui/` 中，构建产物放 `ui/dist/`
