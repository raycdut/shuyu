📊 Request Flow Comparison

Shuyu: without RAG (current) vs. with RAG (proposed)

❌ Without RAG (Current)

Admin Setup (once)
Nothing to configure
RAG Not available. Schema injection is the only mode.
No additional API keys, no vector store, no embedding service.

Step 1
User asks a question
"上个月华东区销量排名"
▼
Step 2
Load full schema from SQLite
load_full_schema() SELECT * FROM imported_tables WHERE database_id=?
All 30+ tables, 450+ columns loaded into memory
▼
Step 3
Build schema prompt
build_schema_prompt() Format every table + column + description
~3,000-5,000 tokens of schema text
▼
Step 4
Inject into agent system prompt
routes/chat.py:203
Agent sees ALL tables. Must infer which are relevant.
▼
Step 5
SQL generation
sql_tool.py LLM reads full schema → generates SQL
⚠️ Risk: LLM picks wrong table (e.g. "suppliers" has a "region" column)
▼
Step 6
Execute SQL → return result
connector.execute() → formatted response
Cost per query
~4,000 tokens for schema alone
Plus LLM generation. Schema-to-response ratio: ~50% waste

✅ With RAG (Proposed)

Admin Setup (once)
Admin enables RAG in settings
/admin → RAG Settings
Toggle ON → select provider (SiliconFlow default) → enter API key
→ Save. Backend auto-builds Tier 1 embeddings for all databases.
→ No restart needed. All subsequent queries use RAG.
— · — · —
Step 1
User asks a question
"上个月华东区销量排名"
▼
Step 2
Route: check RAG enabled?
routes/chat.py state.rag_config.enabled == true
→ proceed to retrieve_schema()
(If enabled == false: fall through to build_schema_prompt as before)
▼
Step 3
Embed question
embed() user_question → vector (1024 dims for BGE-M3)
~100ms, via SiliconFlow BAAI/bge-m3 (or OpenAI)
▼
Step 4
Layered vector search
schema_retriever.retrieve()
← First: Tier 2 (hypothetical questions) — no match (new question)
← Then: Tier 1a (table-level) — orders 0.85, products 0.62
← Then: Tier 1b (column-level) — region 0.78, amount 0.71
→ Top-3: orders, products, regions
→ other_tables: [suppliers, customers, inventory, ...]
▼
Step 5
Build two-part schema prompt
retrieve_schema()
relevant_tables: orders(全列) + products(全列) + regions(全列)
other_tables: suppliers, customers, inventory, ...
~600-900 tokens → 80% less than full schema
▼
Step 6
SQL generation (cleaner prompt)
sql_tool.py LLM sees relevant tables + other_tables list
✅ Less noise, less hallucination risk, faster generation
✅ Can still use other_tables if query requires them
▼
Step 7
Execute SQL → return result
connector.execute() → formatted response
▼
Step 8 (Background)
Self-learning (if enabled)
question_learner.learn()
Feedback gates passed? → Generate 2-3 hypothetical questions
→ Embed + store in Tier 2. Next similar query hits Tier 2 first.
Cost per query
~700 tokens for schema + ~$0.00003 embedding
80-90% schema token reduction. Self-learning adds ~200 tokens per query (background).

Key Observations

  • The RAG path adds admin configuration (one-time) + two extra steps (embedding + vector search) but removes 80-90% of schema tokens
  • Embedding latency (~100ms) is far less than the time saved from a shorter prompt
  • other_tables list prevents information loss — agent knows all available table names
  • Fallback path: if embedding service is down OR admin disables RAG, inject full schema as-is
  • No changes needed to agent loops, SQL execution pipeline, or user-facing UI
  • Self-learning makes the system smarter over time — Tier 2 grows from real usage
